## [Opus-CSO] Sharp Edge #11 v2 派单 · multica WS-218 → CTO Codex (Clash proxy 加 .feishu.cn NO_PROXY)

### 起源

5/22 14:00-15:00 上海时间, Opus-CSO 跟 JC 跑华妆会 5/27 圆桌答案写飞书任务 (`docs +update --doc ShxidIVCDovE1HxfxuPcgkiEnog`), 持续 30+ 分钟 lark-cli MCP transport `POST https://mcp.feishu.cn/mcp EOF`. 同时 Chrome 接管 navigate enterprise 域名 `gdizr0nraj.feishu.cn` 也报 `ERR_CONNECTION_CLOSED`.

Bash 排查发现根因: **Clash proxy 127.0.0.1:7897** 对 feishu 系域名 (CNAME 链 wildcard.feishu.cn.bytedns1.com → all.feishu.cn.w.kunluncan.com) TLS handshake 失败. 跟 5/9 立的 #11 sharp edge 同根因, 但 NO_PROXY 当时没覆盖 feishu / bytedns / kunluncan 三个域名后缀.

### 临时 fix (5/22 已 work)

`unset HTTP_PROXY HTTPS_PROXY ALL_PROXY http_proxy https_proxy all_proxy` → lark-cli docs +update / +fetch 全部恢复. 圆桌答案成功写入飞书 (v2, 5211 字).

### 永久 fix 派单 (WS-218 → CTO Codex)

- 修改 `~/.zshenv` NO_PROXY 加 `.feishu.cn,.bytedns1.com,.kunluncan.com`
- `launchctl setenv NO_PROXY` 同步 GUI 进程
- 重启 multica daemon + Chrome
- Verify: 不 unset proxy 也能跑通 lark-cli
- 沉淀 memory: `feedback_clash_proxy_no_proxy_sharp_edge_0509.md` 加 v2 patch
- 优先级 high (影响 peer-gift / 圆桌写飞书 / 跨员工飞书协作)

### 跨 agent 影响

任何 agent 走 lark-cli 写飞书 / 走 Chrome 操作飞书 enterprise 域名前, 必须 NO_PROXY 已覆盖 `.feishu.cn,.bytedns1.com,.kunluncan.com`. 否则同样撞 EOF 坑.
